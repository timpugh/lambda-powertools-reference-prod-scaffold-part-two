## Production readiness checklist

The hard gates a fork needs to clear before customer traffic touches it. Most items are also broken out individually in the per-service sections below — this section is the at-a-glance summary. The cdk-nag suppressions in [infrastructure/backend_stack.py](infrastructure/backend_stack.py) labeled `"... not needed for sample app"` enumerate the workload-shape gates; the rest are architectural choices outside the nag rule set.

**Workload-shape gates** (cdk-nag suppressions to retire as the gate is closed):

- [ ] Authentication / authorization on the API — currently `AwsSolutions-APIG4` + `AwsSolutions-COG4` suppressed
- [x] API Gateway request validation — a gateway-level `RequestValidator` (`validate_request_body=True`, `validate_request_parameters=True`) is attached to `GET /greeting` in `BackendApp._attach_greeting_resource`; `AwsSolutions-APIG2` suppression retired
- [x] API Gateway throttling (per-stage rate + burst) — implemented via `throttling_rate_limit=100` / `throttling_burst_limit=200` on the Prod stage in `BackendApp`; `Serverless-APIGWDefaultThrottling` suppression retired
- [x] CORS `allow_origin` restricted from `*` to the specific frontend domain — **superseded, not implemented as scoped:** the API moved same-origin behind CloudFront instead (a `/api/*` CloudFront behavior + origin-verify WAF lockdown), so there is no cross-origin browser request left to restrict CORS for at all. Stronger than a restricted `allow_origin` would have been. See [README "Same-origin API and origin lockdown"](README.md#same-origin-api-and-origin-lockdown)
- [x] AWS Backup plan for DynamoDB — **`retain_data`-gated.** `-c retain_data=true` provisions a dedicated `BackupVault` + `BackupPlan` (daily backups kept 35 days, monthly moved to cold storage and kept a year) on top of PITR in `DataStack`. The default (destroy-friendly) shape keeps the prior PITR-only posture and its `DynamoDBInBackupPlan` (NIST/HIPAA) suppressions; PITR is configured to a 1-day rolling window (records TTL out after an hour) with no cross-region/cross-account copies
- [~] DynamoDB deletion protection paired with `RemovalPolicy.RETAIN` — *structure done, one flag from production:* the table and its CMK live in `DataStack`, and `-c retain_data=true` flips both to `RETAIN`, turns on table deletion protection, and enables stack termination protection. Default `false` keeps the template destroy-friendly. The remaining gate is simply *setting* `retain_data=true` for a production deployment
- [x] Lambda reserved concurrency — implemented via `reserved_concurrent_executions=100` on `ApiFunction`; `NIST.800.53.R5-LambdaConcurrency` / `HIPAA.Security-LambdaConcurrency` suppressions retired

**Edge and transport gates:**

- [ ] Custom domain + ACM certificate on CloudFront and API Gateway (the defaults `*.cloudfront.net` and `*.execute-api.amazonaws.com` pin the TLS floor at TLS 1.0)
- [x] CloudFront `Strict-Transport-Security` header — added via a custom `ResponseHeadersPolicy` (replacing the managed `SECURITY_HEADERS` policy) with HSTS (`max-age=31536000`, `includeSubDomains`; `preload` left off until a custom domain exists) and a CSP
- [x] CloudFront-bypass window on the `execute-api` URL closed — a REGIONAL WAF WebACL (four managed threat rule groups plus a `RejectNonCloudFront` rule) is associated with the API Gateway Prod stage. CloudFront injects a Secrets-Manager-generated `x-origin-verify` header on every origin request, and the WAF rule blocks any request that doesn't byte-match it — the regional ACL now **rejects** non-CloudFront callers outright (fail-closed 403) rather than merely inspecting them. See [README "Same-origin API and origin lockdown"](README.md#same-origin-api-and-origin-lockdown)

- [~] **[SLSA](https://github.com/slsa-framework/slsa) build provenance** — prototyped (Build L3 via [slsa-github-generator](https://github.com/slsa-framework/slsa-github-generator)) and **deliberately removed**: this repo is a fork-this template whose deploys rebuild from source, so no released artifact is ever consumed, and wiring it forced the only non-SHA-pinned action in the repo. Revisit only in a fork that publishes a genuinely consumed artifact (shared Lambda layer, wheel, container). Rationale in README "Code scoring and attestation".
- [~] **[SonarQube](https://www.sonarsource.com/products/sonarqube/) scanning + badges** — **not pursued.** Sonar has no GitHub-native mode: the scanner needs a Sonar server, and that server ingests the source. Cloud (SaaS) = third-party code access (the Codecov posture again); self-hosted = running and publicly exposing a server just for a README badge. Sonar's value is already covered in-house and inside GitHub — ruff (incl. flake8-bandit `S`) + bandit (SAST), mypy (types), pylint (design smells), CodeQL (dataflow/taint), radon + xenon (complexity). A fork that already runs a SonarQube server can point the scanner at it. Rationale in README "Code scoring and attestation".
- [~] **[Codecov](https://github.com/codecov/codecov-action) coverage tracking** — **not pursued.** Codecov is a third-party SaaS (the uploader ships the report off-platform and Codecov pulls the source via a granted token to render it; its uploader was the 2021 supply-chain breach vector), which conflicts with this repo's keep-the-build-inside-GitHub posture. Coverage is kept in-house: the CI artifact, the enforced 100% `lambda/` gate, `make coverage`'s combined HTML, and an in-house **coverage badge** (`make coverage-badge` → shields endpoint JSON published to our own Pages by `docs.yml`). A fork wanting hosted history + PR-diff comments can still wire Codecov. Rationale in README "Code scoring and attestation".
- [~] **Fuzzing** ([OpenSSF Scorecard](https://github.com/ossf/scorecard/blob/main/docs/checks.md#fuzzing) `Fuzzing: medium`) — **not pursued.** Fuzzing pays off in proportion to hand-rolled parsing / deserialization / state-machine / regex / memory-unsafe code, and this codebase has almost none of it in *its own* code: one route (`GET /greeting`) that accepts no query/path/body parameters, and the only bespoke untrusted-input handling is defensive header-key lowercasing plus `_resolve_tenant_id`'s `dict.get(...)` + `str()` fallback — nothing with a parser to crash. Everything that actually parses untrusted bytes (JSON decode, the Powertools resolver, JMESPath, Pydantic validation) lives in upstream libraries already fuzzed at the source (pydantic-core is Rust with its own fuzzing). The realistic path to the Scorecard badge for Python is **ClusterFuzzLite + Atheris** harnesses in a new CI lane (OSS-Fuzz is aimed at widely-depended-upon libraries and is a poor fit for a fork-this template); that adds a native-toolchain fuzzing pipeline and corpus for near-zero yield, cutting against the same keep-the-build-inside-GitHub / minimal-pinned-surface posture that dropped SLSA / Sonar / Codecov above. **Revisit when a route is added that accepts a real request body or does custom parsing/decoding** — wire it then, targeted at that specific handler. (A lighter middle ground — Hypothesis property-based tests over the input-normalization helpers — lives in the existing pytest suite but likely won't move the Scorecard check, whose Python detector is Atheris, and would mostly re-confirm the unit tests given how defensive that code already is.)

## Security

- [ ] **API Gateway authentication** — add an API key, IAM auth, or Cognito authorizer to restrict access
- [x] **Lambda least-privilege IAM** — DDB, SSM, AppConfig grants are scoped to the specific resource ARNs. `appconfig:GetLatestConfiguration` and X-Ray segment publishes remain at `Resource: "*"` because their target ARNs (configurationsession token, segment) are dynamically generated at call time and not addressable by IAM at policy-creation time — documented in the IAM5 nag suppression.
- [ ] **VPC placement** — place the Lambda function inside a VPC if it needs to access private resources
- [x] **CORS origin restriction** — **superseded, not implemented as scoped.** The Lambda handler no longer configures a `CORSConfig` at all: the API moved same-origin behind CloudFront (a `/api/*` CloudFront behavior with a viewer-request rewrite function, plus the origin-verify WAF lockdown below) instead of restricting `allow_origin` to the CloudFront domain. Same-origin is a stronger posture than a restricted origin would have been — there's no cross-origin request left to misconfigure CORS for. See [README "Same-origin API and origin lockdown"](README.md#same-origin-api-and-origin-lockdown).
- [x] **CloudFront Strict-Transport-Security header** — done. A custom `ResponseHeadersPolicy` in [infrastructure/frontend_stack.py](infrastructure/frontend_stack.py) replaces the AWS-managed `SECURITY_HEADERS` policy, reproducing its four headers (X-Content-Type-Options, X-Frame-Options, Referrer-Policy, X-XSS-Protection) and adding HSTS (`max-age=31536000`, `include_subdomains=True`, `preload=False`) plus a CSP (see "CSP header on the static page" below — both now ship from this one policy). `preload` is intentionally off until a custom domain exists, since the preload list is keyed on the registrable domain, not a shared `*.cloudfront.net` host.
- [x] **Narrow the CDK bootstrap permissions** — **done.** The default `cdk bootstrap` creates a `CloudFormationExecutionRole` with `AdministratorAccess`; any identity that can `sts:AssumeRole` into the deployment roles (by default, any principal in the account) could do anything in the account during deploy. Fixed via the `cdk-scaffold-boundary` managed policy ([infrastructure/bootstrap/cdk-scaffold-boundary.json](infrastructure/bootstrap/cdk-scaffold-boundary.json), deployed with `make bootstrap-boundary`), then re-bootstrapping with `cdk bootstrap --custom-permissions-boundary cdk-scaffold-boundary` (which attaches the boundary to the `CloudFormationExecutionRole` only, not the other bootstrap roles — see the item above) — CFN can do anything inside the boundary's allow-list but can't use it to create unbounded principals or grant itself admin past it: it can't create roles/users without the boundary, can't replace the boundary on an existing principal, can't attach or inline a policy onto a principal that doesn't already carry the boundary (`DenyPolicyEditsOnUnboundedPrincipals`, added alongside the create/replace denies), and can't tamper with the boundary policy itself. This shipped **before** the Deploy workflow above got deploy-capable credentials, per this item's original sequencing note. The same boundary also applies to runtime roles, not just deployment roles — see the IAM section's [Permissions boundary on the Lambda execution role](#iam) cross-reference. **Documented residual:** `iam:UpdateAssumeRolePolicy` on unbounded roles remains allowed inside the boundary — pair with org-level SCPs for defense in depth. **Still open at the org level:** SCPs via AWS Organizations to prevent tampering with the boundary, and restricting who can assume `DeploymentActionRole` to the CI role + named humans — this repo is single-account/solo-dev, so that layer isn't wired. See [README "CI/CD pipeline"](README.md#cicd-pipeline).
- [ ] **Branch protection enforced, not routinely bypassed** — branch protection rules exist on `main` (required status checks from CI), but `git push` reports `Bypassed rule violations for refs/heads/main`, meaning maintainers routinely override the gate. For production, each bypass should be an audited exception with a recorded rationale, not a normal merge workflow — otherwise the gate exists only to slow down non-maintainers. Tighten via GitHub branch protection settings: require status checks to pass for *administrators* too, require pull-request reviews, and dismiss stale reviews on new commits. The signal "maintainer pushes directly to main" should be unusual enough that each occurrence triggers a question.
- [ ] **Enforce TLS 1.2+ minimum on both edges** — the CloudFront distribution and API Gateway both currently sit on AWS-managed default certificates (`*.cloudfront.net` and `*.execute-api.{region}.amazonaws.com`), which pin the TLS floor at **TLS 1.0**. Verified empirically: `curl --tls-max 1.0 https://<dist>.cloudfront.net` and the equivalent against the execute-api endpoint both complete a full handshake. The CDK code in [frontend_stack.py](infrastructure/frontend_stack.py) sets `TLS_V1_2_2021` but AWS silently overrides it whenever `CloudFrontDefaultCertificate: true`. The cdk-nag rule `AwsSolutions-CFR4` correctly flags this and is intentionally suppressed at the stack level. **Fix path:** acquire a domain, provision an ACM certificate (CloudFront cert must live in us-east-1, API Gateway custom-domain cert lives in the API's region), attach as `viewer_certificate` / `apigateway.DomainName`, then set the strongest matching `securityPolicy` (e.g. `SecurityPolicy_TLS13_1_3_2025_09` for a regional API Gateway domain, and `TLSv1.2_2021` minimum on CloudFront). Once the custom domain is wired and verified, remove the CFR4 suppression. **Update:** the API is now `REGIONAL`, not `EDGE` — CloudFront already fronted it, so the redundant edge layer was removed (see [README "Same-origin API and origin lockdown"](README.md#same-origin-api-and-origin-lockdown)), which already unlocks the regional `securityPolicy` set (including post-quantum and PFS variants) for the custom-domain work above whenever it happens.

When a second Lambda/service is added, restructure the runtime source so each service owns its own directory instead of piling handlers into one `lambda/`: e.g. `services/service_a/` (its `app.py`, `requirements.txt`, tests) alongside `services/service_b/`. Each service then gets its own CDK construct (mirroring `BackendApp`) whose `PythonFunction(entry="services/service_a", …)` points at that directory, so per-service runtime deps and the `check-lock` / `compare-openapi` drift gates stay isolated per service. Keep the single-`lambda/` layout while there's one function — the split only pays off at the second.
- [ ] **Input validation on caller-facing inputs** — `enable_validation=True` is set on the `APIGatewayRestResolver` in [lambda/app.py](lambda/app.py) and Pydantic models drive response validation, so the framework is wired. The `/greeting` route currently accepts no query string, path, or body parameters, so there is nothing to validate yet. When new routes are added that accept caller input, type-annotate the handler parameters with Pydantic-compatible types (or `Annotated[..., Query/Body]`) so Powertools enforces the schema and rejects malformed input with a 422 before any business logic runs.
- [ ] **Contributing guide** — `CONTRIBUTING.md` with fork/branch/PR workflow and pre-commit setup instructions
- [x] **Changelog** — `CHANGELOG.md` auto-generated from conventional commit history via [git-cliff](https://github.com/orhun/git-cliff), configured in `cliff.toml`. Regenerate after each release with `git cliff -o CHANGELOG.md`. Dependabot bumps and merge commits are filtered out so the changelog reflects feature/fix/docs/CI history rather than dependency churn.
- [x] **`cdk destroy` BucketDeployment cache-invalidation race** — Fixed by decoupling cache invalidation from `s3deploy.BucketDeployment` into a standalone `cr.AwsCustomResource` that defines `on_create` + `on_update` only (no `on_delete`). CFN removes the resource from stack state during teardown without making any CloudFront API call to race with. CallerReference is gated on the BucketDeployment's content-hashed S3 object key so invalidations only fire when frontend assets actually change, conserving the 1000-paths/month free quota. Permanent fix for [aws/aws-cdk#15891](https://github.com/aws/aws-cdk/issues/15891) — see [README "Decoupled CloudFront cache invalidation"](README.md#design-decisions-and-known-limitations) for the longer write-up.

## Service-level hardening

Per-service hardening items grouped by AWS service so each block can be tackled in isolation. Items deferred for "no direct CDK construct" or "would require restructuring upstream data" reasons are flagged.

### API Gateway

- [x] **APIGW-layer request validation** — a gateway-level `RequestValidator` (`validate_request_body=True`, `validate_request_parameters=True`) is attached to `GET /greeting` in `BackendApp._attach_greeting_resource`, rejecting malformed requests at the gateway before they reach Lambda billing; `AwsSolutions-APIG2` retired. Validation still also runs inside Lambda via `enable_validation=True` on the resolver — belt-and-suspenders, and the layer that matters once a parameterized route exists.
- [ ] **Resource policy on the REST API** — restrict invocation to specific source IPs, VPCs, or AWS accounts without needing a full authorizer. Useful for partner-facing or internal-only APIs.
- [x] **Close the CloudFront-bypass window on `execute-api.{region}.amazonaws.com`** — *both options (a) and (b) done.* A REGIONAL WAF WebACL is associated with the Prod stage in [infrastructure/backend_app.py](infrastructure/backend_app.py) (`_attach_regional_waf`), mirroring the four managed threat rule groups from the CLOUDFRONT-scoped ACL via the shared `build_managed_threat_rules` helper in [infrastructure/nag_utils.py](infrastructure/nag_utils.py). Option (b), the full lockdown, now rides alongside it: CloudFront injects a Secrets Manager secret as the `x-origin-verify` origin header on every CloudFront→origin request, and a `RejectNonCloudFront` WAF rule blocks any request that doesn't byte-match it exactly — the origin now **rejects** direct callers with a flat `403` instead of merely inspecting them (there is no rotation Lambda by design; rotation is manual — put a new secret value, then redeploy. See [README "Same-origin API and origin lockdown"](README.md#same-origin-api-and-origin-lockdown) for the full mechanism, the deploy-order 403 window, and the rotation recipe). `AwsSolutions-APIG3` and the NIST/PCI `APIGWAssociatedWithWAF` suppressions are retired.
- [~] **Custom domain + endpoint export + Mutual TLS** — out of scope for the sample app: custom domain depends on owning a domain ([TLS item](#security)); MTLS only justifies its complexity once you have partners. *Canary deployments are now implemented* (CodeDeploy traffic shifting on the Lambda alias — see the Lambda section below).

### Lambda

- [ ] **Dead Letter Queue (DLQ)** — already covered by [Observability](#observability). Lower priority for the synchronous API path; required if you ever add async invokes (EventBridge, SNS, S3 events).
- [x] **Reserved concurrency** — `reserved_concurrent_executions=100` is set on `ApiFunction` in [infrastructure/backend_app.py](infrastructure/backend_app.py), capping how much of the account concurrency pool a runaway loop or spike on `/greeting` can consume. `NIST.800.53.R5-LambdaConcurrency` / `HIPAA.Security-LambdaConcurrency` suppressions retired (the same rule IDs in `CDK_LAMBDA_SUPPRESSIONS` stay — they cover CDK-managed singleton Lambdas whose concurrency isn't configurable).
- [x] **Async failure destination on the AwsCustomResource provider Lambda** — implemented via `attach_async_failure_destination()` in [infrastructure/nag_utils.py](infrastructure/nag_utils.py). The provider singleton (which CloudFormation invokes asynchronously during stack lifecycle events) writes failed invocation envelopes to an SQS DLQ (CMK-encrypted, 14-day retention, SSL-enforced) so post-mortem evidence isn't lost when a CR crashes through Lambda's two automatic async retries. The ApiFunction itself remains synchronous (API Gateway proxy) and intentionally has no DLQ.
- [x] **Canary deployments (CodeDeploy traffic shifting)** — the API integrates with a `live` Lambda alias and a `codedeploy.LambdaDeploymentGroup` shifts traffic onto each new version with automatic rollback on a canary alias-errors alarm (`_attach_canary_deployment` in [infrastructure/backend_app.py](infrastructure/backend_app.py)). Environment-gated: `CANARY_10PERCENT_5MINUTES` in prod, `ALL_AT_ONCE` in dev/ephemeral. See [README "Deployment safety"](README.md#deployment-safety-canary-lambda--appconfig-rollback).
- [ ] **SnapStart for Python** — Python SnapStart launched in November 2024. Roughly 70% cold-start reduction at the cost of a one-time init snapshot. Worth enabling if cold-start latency becomes a UX issue.
- [x] **Lambda Insights** — extension-based enhanced metrics (CPU time, memory utilization, init duration, network bytes) enabled via `insights_version=lambda.LambdaInsightsVersion.VERSION_1_0_498_0` on `ApiFunction`; ~$0.50/month for the metric stream. Its destination log group, `/aws/lambda-insights`, is regional and shared account-wide — deliberately **not** stack-owned or targeted by this template's dangling-resource cleanup (claiming/deleting a shared-name regional group would collide with any other Insights user in the account).

### DynamoDB

The table and its dedicated CMK live in their own stack, [`DataStack`](infrastructure/data_stack.py), so their lifecycle is independent of the stateless compute (CDK best practice: keep stateful resources in their own stack). See [README "Stateful data stack and `retain_data`"](README.md#stateful-data-stack-and-retain_data).

- [~] **Deletion protection + `RemovalPolicy.RETAIN`** — *wired behind one flag.* `deletion_protection=True`, `RemovalPolicy.RETAIN` (table + CMK), and stack termination protection all ride the `retain_data` switch (`-c retain_data=true`, plumbed `app.py` → `AppStage` → `DataStack`). Default `false` keeps the idempotency cache (regenerable data) destroy-friendly for dev/ephemeral environments; a production fork sets it `true`.
- [x] **AWS Backup plan** — **`retain_data`-gated.** `-c retain_data=true` provisions a dedicated `BackupVault` (this stack's CMK, `RemovalPolicy.RETAIN`) and a `BackupPlan` with daily backups kept 35 days plus a monthly backup moved to cold storage and kept a year, on top of PITR (which alone caps at 35 days; this table configures its 1-day minimum to match the 1-hour TTL). The default (destroy-friendly) shape stays PITR-only, and the `DynamoDBInBackupPlan` nag suppressions on `DataStack` apply only there — they're retired in the `retain_data=true` shape.

### S3

- [x] **Versioning on the frontend bucket** — enabled, with a 30-day noncurrent-version expiry lifecycle rule so old versions don't accumulate indefinitely. Git is still the source of truth for deployed assets, but versioning now gives in-bucket recovery too if assets are ever manually overwritten, and it's also a prerequisite for cross-region replication.
- [ ] **S3 Inventory / Storage Lens / Object Lock / Macie** — `(Required)` per the broader S3 best-practice set: Inventory exports object-level metadata daily, Storage Lens gives org-wide visibility, Object Lock enforces write-once retention for compliance, Macie scans for sensitive data. None justify themselves at sample-app scale; revisit at production scale or under compliance scope.

### CloudFront

- [ ] **Scope down the OAC SSE-KMS key-policy wildcard after first deploy** — `S3BucketOrigin.with_origin_access_control()` on the KMS-encrypted frontend bucket emits a key-policy condition whose `aws:SourceArn` matches *every* distribution in the account (`arn:aws:cloudfront::<account>:distribution/*`), to break the circular dependency between the key, bucket, and distribution on the *initial* deploy — CDK flags this as `@aws-cdk/aws-cloudfront-origins:wildcardKeyPolicyForOac` (a synth/deploy warning, observed on a live deploy). Once the distribution exists, AWS recommends tightening the condition to the specific distribution ARN. Doing so in CDK requires a second-deploy override (or a post-deploy key-policy edit) because the distribution ID isn't known at first synth. Low blast radius for a single-distribution account — only this account's CloudFront service can use the key, and only for this stack's bucket — but worth scoping for a multi-distribution account or a strict least-privilege posture.

### Athena

- [x] **Bytes-scanned-per-query data usage control** — `bytes_scanned_cutoff_per_query=1 GiB` is set on the access-logs workgroup in [infrastructure/frontend_stack.py](infrastructure/frontend_stack.py); Athena cancels any query that would exceed it, capping per-query cost against a forgotten `WHERE`. Generous for these log tables at 7-day retention — raise it in a fork with larger log volumes.
- [ ] **`MinimumEncryptionConfiguration` on the workgroup (belt-and-suspenders)** — per the [Athena minimum-encryption docs](https://docs.aws.amazon.com/athena/latest/ug/workgroups-minimum-encryption.html), the dedicated `MinimumEncryptionConfiguration` field enforces a floor *even when client overrides are allowed*. CDK 2.248's L1 `CfnWorkGroup` doesn't expose that field directly; achieving it requires a property override on the underlying CFN resource. The current setup (`enforce_work_group_configuration=True` + `EncryptionOption=SSE_KMS`) already prevents per-query overrides at this workgroup specifically, which is project-scoped and not account- or region-wide, so for a sample app it's equivalent. **Checked 2026-07-08:** still not supported — neither the pinned CDK L1 (`CfnWorkGroup.WorkGroupConfigurationProperty`) nor the current `AWS::Athena::WorkGroup` CloudFormation schema exposes the field (verified against the CFN Template Reference), so a property override would be rejected at deploy; the workgroup-enforced `SSE_KMS` posture stands. Re-check when bumping `aws-cdk-lib`.
- [ ] **Workgroup-level query result reuse** — *Deferred: no CDK / CFN support.* The CFN `AWS::Athena::WorkGroup.WorkGroupConfiguration` schema does not expose a result-reuse default. Result reuse can only be set per-query in `StartQueryExecution` calls, which doesn't fit a CDK-declared workgroup. Revisit when CFN adds `ResultReuseConfiguration` to the workgroup schema.
- [ ] **Cost allocation tags on the workgroup** — apply tags (Environment, Project, Owner) so Athena query costs roll up cleanly in Cost Explorer.
- [ ] **Partition projection on access-log tables** — *Deferred: requires upstream log restructuring.* The Glue tables for `cloudfront_logs` and `s3_access_logs` currently scan every file in the prefix on every query because CloudFront standard v1 logs and S3 server access logs both write to flat key spaces (no `year=YYYY/month=MM/...` directories). Implementing partition projection requires either: (a) migrating CloudFront from standard v1 to standard v2 logs (separate CDK construct via the Logs delivery API) AND switching S3 server access logs to `target_object_key_format=PartitionedPrefix` for date-based S3 prefixes; or (b) a re-organize Lambda to copy logs into Hive-style partition prefixes. Both are significant changes and (a) requires CDK to catch up on standard v2 wiring. Revisit when the v2 path is well-supported.
- [~] **Athena CloudWatch alarms** — `AthenaFailedQueriesAlarm` (`FrontendStack._attach_analytics_alarms`) ships, alarms on ≥3 FAILED DML queries/hour in the access-logs workgroup. A companion `ProcessedBytes` alarm (sum per workgroup over a threshold) remains open — the metric exists in AWS/Athena with the same dimension set (`WorkGroup`, `QueryState`, `QueryType`) that the shipped alarm already uses, so adding it follows the same pattern in `infrastructure/frontend_stack.py` as a straightforward extension. The existing `bytes_scanned_cutoff_per_query=1 GiB` workgroup setting (above) is a separate preventive per-query quota that cancels queries exceeding the limit, distinct from after-the-fact alerting on historical bytes scanned.

- [~] **Glue Data Catalog encryption** — *implemented and deliberately reverted.* Two reasons: (1) `AWS::Glue::DataCatalogEncryptionSettings` is account/region-scoped — there is one Glue catalog per account per region, so deploying this reference architecture into an account with other Glue users would silently override their encryption settings or conflict outright; (2) the catalog metadata in *this* stack (column names from public CloudFront/S3 access-log schemas) carries no confidentiality requirement, and the stack has no Glue connections, so encrypting connection passwords protects nothing. If you fork this and your catalog will hold genuinely sensitive table metadata, put `glue.CfnDataCatalogEncryptionSettings` into a separate, intentionally account-scoped stack so the deploy boundary reflects the resource's account-wide nature. See the Glue Data Catalog write-up in the README for the longer rationale.

### WAF

- [x] **CloudWatch alarms on BlockedRequests spikes** — `BackendApp._attach_waf_alarms` alarms on `AWS/WAFV2` `BlockedRequests` (> 100 / 5 min) for both the regional (API Gateway) and CloudFront WebACLs, addressed by their CloudWatch metric *name* (`VisibilityConfig.MetricName`) with no cross-stack/cross-region reference. The CloudFront alarm omits the `Region` dimension (the one protected-resource type that does) and is only created when the stack is in `us-east-1`, where CloudFront WAF metrics publish. Routes through the shared `route_operational_alarm` helper — prod alarm topic, dashboard-only + acknowledged suppressions in non-prod.
- [x] **WebACLCapacityUnits (WCU) alarm — investigated, not implementable.** WAFv2 does not publish a WCU-consumption CloudWatch metric at all; WCU is a **static** per-rule capacity number fixed at WebACL-authoring time (visible in the console/`GetWebACL`), not a runtime signal that accrues traffic-dependent usage. There is nothing to alarm on. The actionable equivalent is a one-time synth-time check (or manual review) of the WebACL's total declared WCU against the 1500 WCU pricing threshold when adding rules — not a CloudWatch alarm.
- [x] **WAF logs to S3** — both WebACLs (CloudFront-scoped in `WafStack`, regional in `_attach_regional_waf`) deliver access logs to `aws-waf-logs-*` S3 buckets (SSE-S3, 90-day lifecycle) instead of CloudWatch — cheaper long-term retention, queryable via Athena. WAF owns the bucket policy (CDK pre-declares the delivery grant + orders the logging config after it; verified on a live deploy). Built via `create_waf_logs_bucket` in [infrastructure/nag_utils.py](infrastructure/nag_utils.py).
- [x] **WAF Glue/Athena tables** — two partition-projected Glue tables (`waf_cloudfront_logs`, `waf_regional_logs`) over the `aws-waf-logs-*` buckets, plus 8 Athena named queries (recent blocked / top blocked IPs / top terminating rules / by country, per WebACL), restore the WAF log analysis the WAF→S3 switch retired. Built in the frontend stack's Glue DB + Athena workgroup; the Stage computes the WAF log S3 locations (no cross-stack ref). Schema + partition projection follow AWS's documented WAF Athena DDL. See [README "Audit stack and log retention"](README.md#audit-stack-and-log-retention).
- [x] **WAF logging — `redacted_fields` and `logging_filter`** — both WebACLs (CloudFront in `WafStack`, regional in `BackendApp`) now redact the `Authorization` and `Cookie` headers (`waf_log_redacted_fields()` in `nag_utils.py`) and drop ALLOW-verdict records (`WAF_LOG_DROP_ALLOW_FILTER`), via a shared helper/constant pair so the two `CfnLoggingConfiguration`s never drift apart. Log volume now scales with threat traffic (BLOCK/COUNT/CAPTCHA/CHALLENGE only), not with legitimate traffic.
- [ ] **Pin AMR managed-rule-group versions** — the WebACL currently uses the floating "default" version of each AMR. Pinning to a specific version (e.g. `Version_2.0` of `AWSManagedRulesCommonRuleSet`) means rule updates from AWS go through your release process rather than landing automatically. Trade-off: less drift, but you have to track AMR change announcements and bump the version manually.
- [ ] **Subscribe to AMR SNS update topics** — AWS publishes notifications when managed rule groups change behavior (action shifts, new sub-rules, deprecations). Subscribing means you find out *before* the change lands rather than from a Slack page about a sudden block-rate spike.
- [x] **Add `AWSManagedRulesAnonymousIpList` AMR** — implemented at WebACL priority 3 in [waf_stack.py](infrastructure/waf_stack.py). Blocks Tor exit nodes, hosting providers, and known anonymizing services.
- [ ] **CAPTCHA / Challenge actions on high-risk routes** — N/A until login/signup/credential-bearing routes exist. When they do, replacing a `Block` action on a rate-based rule with `Challenge` (silent JS challenge) or `CAPTCHA` (visible) catches bots without false-positiving real users.
- [ ] **Geo-blocking rule** — if you have countries that should never reach the app, a single `geoMatchStatement` rule blocks them at the edge. Free, low WCU. Skip if global traffic is expected.
- [ ] **Bot Control / ATP / ACFP** — paid advanced AMRs (Bot Control common ~$10/M requests, targeted higher). Bot Control covers automated browser-based traffic; ATP (Account Takeover Prevention) protects login flows; ACFP (Account Creation Fraud Prevention) protects signup flows. Skip until login/signup routes exist and are observably under attack.
- [~] **AntiDDoSRuleSet** — *implemented and deliberately reverted.* The rule group provides L7 anti-DDoS via Challenge + Block, but it carries a $20/month per-WebACL entity activation fee plus $0.15/million requests on top of the standard WAF base ($5/month WebACL + $1/month per rule + $0.60/million inspections). For this reference architecture that would have raised fixed WAF cost from $10/month to $31/month for one rule whose Block arm only fires on AWS-observed high-confidence DDoS classification, which the existing forwarded-IP rate limit (200 req/5min) already covers at the threat profile a Serverless App demo realistically faces. Also requires `ManagedRuleGroupConfig.ClientSideActionConfig` declared and (when `UsageOfAction=ENABLED`) at least one URI in `ExemptUriRegularExpressions` — neither fits a CloudFront-fronted SPA with no health-check or m2m endpoints to exempt. See the AntiDDoS write-up in the README for the longer rationale. Revisit if the workload ever sees enough traffic that AWS's classifier has signal to act on, or when the $20/month is rounding error.

### AppConfig

- [ ] **Gradual deployment + alarm rollback monitor** (*built, opt-in via `-c appconfig_monitor=true`; off by default*) — the gradual strategy (LINEAR 25%/step over 10 min, 5-min bake) + CloudWatch alarm monitor that auto-rolls-back a bad flag rollout is wired in [`infrastructure/backend_app.py`](infrastructure/backend_app.py) (`_attach_appconfig_rollback_monitor`), gated behind the `appconfig_monitor` context flag. The observable signal always ships: the handler emits a `FeatureFlagEvaluationFailure` EMF metric on the feature-flag fallback path (a bad config is caught by the fallback and returns 200, so it produces no Lambda error — that custom metric is the only signal). **Why it's off by default:** an AppConfig deployment monitor rolls back when its alarm is in `ALARM` **or `INSUFFICIENT_DATA`** ([monitoring-deployments.html](https://docs.aws.amazon.com/appconfig/latest/userguide/monitoring-deployments.html)). On a cold CFN stack the metric has never reported and a fresh alarm starts `INSUFFICIENT_DATA`, so AppConfig aborts the very first deployment — the stack can never reach `CREATE_COMPLETE` (proven on a live deploy, even with `treat_missing_data=NOT_BREACHING`). **Adoption sequence (do NOT set the flag on the cold deploy):** (1) `make deploy` with the default (all-at-once, no monitor) so the stack reaches `CREATE_COMPLETE` and the Lambda starts reporting the metric; (2) `make deploy-appconfig-monitor` (guarded — refuses unless the stack already exists in an updatable `*_COMPLETE` state) for ongoing flag changes. To persist it across plain deploys, also set `"appconfig_monitor": true` in `cdk.json` after step 1. See [README "Deployment safety"](README.md#deployment-safety-canary-lambda).
- [x] **AWS AppConfig Lambda extension layer** — implemented. `BackendApp._attach_appconfig_extension_layer` wires the AWS-published extension (ARM64, region-pinned ARN; `us-east-1` pins layer version 2.0.18836) onto the function, prefetched at cold start via `AWS_APPCONFIG_EXTENSION_PREFETCH_LIST`. The handler's feature-flag store (`lambda/extension_store.py`'s `AppConfigExtensionStore`) reads cached flag data from the extension's local `http://localhost:2772` endpoint instead of calling the AppConfig data plane via `boto3`/`AppConfigStore` directly — cutting both AppConfig API spend and cold-start latency, per [Using AWS AppConfig Agent with AWS Lambda](https://docs.aws.amazon.com/appconfig/latest/userguide/appconfig-integration-lambda-extensions.html). The failure path is unchanged: any extension-endpoint failure raises `ConfigurationStoreError`, handled by the existing fallback (default flag values + `FeatureFlagEvaluationFailure` EMF metric).

### CloudWatch RUM

- [ ] **Lower the RUM session sample rate at scale** — currently `session_sample_rate=1.0` (100%). Per the [RUM authorization docs](https://docs.aws.amazon.com/AmazonCloudWatch/latest/monitoring/CloudWatch-RUM-get-started-authorization.html) and [pricing](https://aws.amazon.com/cloudwatch/pricing/), RUM is billed per ingested event so 100% sampling becomes expensive once real traffic hits the page. 5–10% is a common production starting point. Also revisit `allow_cookies` and `enable_x_ray` against the workload's privacy posture — both are on for full session attribution today.
- [x] **Bound RUM ingestion cost against guest-pool abuse** — the RUM identity pool is `allow_unauthenticated_identities=true` (inherent to browser RUM — guests need credentials to call `rum:PutRumEvents`), so anyone can mint guest credentials via `cognito-identity:GetId` / `GetCredentialsForIdentity` and POST events to the RUM data plane directly. RUM is billed per ingested event and there is no per-caller server-side cap. **`session_sample_rate` does not mitigate this** — it is a client-side knob the RUM *browser* client honors; a caller hitting `PutRumEvents` directly ignores it entirely (lowering it only reduces *legitimate* traffic, above). The public guest pool can't be eliminated without breaking browser RUM, so the guardrail is **detection**, implemented two ways: a `RumSessionSpikeAlarm` (`FrontendStack._attach_analytics_alarms`, `AWS/RUM` `SessionCount` > 1000/hour) as a volume proxy, and a prod-only [AWS Budgets](https://docs.aws.amazon.com/cost-management/latest/userguide/budgets-managing-costs.html) spend budget (`BackendApp._attach_cloudwatch_spend_budget`, $10/month reference limit on the `AmazonCloudWatch` cost-filter service, which RUM bills under) notifying the alarm topic at 80% actual spend. **Needs live verification:** Cost Explorer wasn't queryable at implementation time to confirm the `AmazonCloudWatch` service-filter string against a real bill — a wrong filter value fails silently (the budget still deploys and just tracks $0 forever). The existing `bytes_scanned_cutoff_per_query` (Athena) and API Gateway stage throttling guards do not cover this path; these two alarms do.

### CloudTrail

- [ ] **Multi-region management trail** — the trail in this stack is regional and S3-data-events-only by design (it captures every Get/Put/Delete against the audited buckets). Per [WKLD.07 of the AWS Startup Security Baseline](https://docs.aws.amazon.com/prescriptive-guidance/latest/aws-startup-security-baseline/wkld-07.html), production accounts should also have a separate management-event multi-region trail capturing IAM/STS/KMS/EC2 calls. Typically deployed at the org/account level (Control Tower / Landing Zone) rather than per-app — see the [Account-level identity governance](#iam) item under IAM.
- [ ] **Log file integrity validation already on; consider CloudTrail Lake** — `enable_file_validation=True` is set, which produces signed digest files. For richer querying than Athena over flat CloudTrail JSON, [CloudTrail Lake](https://docs.aws.amazon.com/awscloudtrail/latest/userguide/cloudtrail-lake.html) gives SQL-queryable event stores with up to 7-year retention. Trade-off: per-event ingestion cost vs. the current ~free S3-as-archive path. Worth flagging once compliance scope (HIPAA, PCI) actually applies.

### Frontend / browser

- [~] **CSP header on the static page** — *baseline done, hardening open.* A `Content-Security-Policy` now ships from the custom `ResponseHeadersPolicy` in [infrastructure/frontend_stack.py](infrastructure/frontend_stack.py) (paired with HSTS in [Security](#security), from the same policy). It locks `default-src`/`object-src 'none'`/`base-uri`/`frame-ancestors 'none'` and scopes `script-src`/`connect-src` to the exact AWS endpoints (RUM client + data plane, Cognito Identity). The API call no longer needs a `connect-src` entry at all now that it's same-origin behind CloudFront (see [README "Same-origin API and origin lockdown"](README.md#same-origin-api-and-origin-lockdown)) — one fewer origin to keep pinned. It still carries `'unsafe-inline'` on `script-src`/`style-src` because `frontend/index.html` ships inline `<script>`/`<style>` blocks (the RUM bootstrap shares state with the body handler via synchronous head execution). To reach a strict nonce/hash CSP, externalize the two inline scripts into `'self'`-served `.js` files first — CloudFront can't mint per-response nonces and static hashes break on every edit.
- [ ] **No SRI on the RUM client script** — `frontend/index.html` loads `https://client.rum.us-east-1.amazonaws.com/1.x/cwr.js` over a major-version-floating URL so AWS-published security patches reach the browser without a redeploy. The trade-off: a fixed `integrity=` SRI hash would break on the first patched release. The trust model here is "AWS-served domain over TLS 1.2+" rather than pinned content hash. If your fork pins to a specific patch version (e.g., `1.21.0`), add the matching SRI hash so the browser refuses tampered scripts; otherwise document the trade-off explicitly so a future review doesn't re-flag the absence of `integrity=`.
