## Security

- [ ] **API Gateway authentication** ‚Äî add an API key, IAM auth, or Cognito authorizer to restrict access
- [x] **Lambda least-privilege IAM** ‚Äî DDB, SSM, AppConfig grants are scoped to the specific resource ARNs. `appconfig:GetLatestConfiguration` and X-Ray segment publishes remain at `Resource: "*"` because their target ARNs (configurationsession token, segment) are dynamically generated at call time and not addressable by IAM at policy-creation time ‚Äî documented in the IAM5 nag suppression.
- [ ] **VPC placement** ‚Äî place the Lambda function inside a VPC if it needs to access private resources
- [ ] **CORS origin restriction** ‚Äî the Lambda handler uses `allow_origin="*"`. In production, restrict to the specific CloudFront domain and set `allow_credentials=True` if cookies or Authorization headers are needed.
- [x] **CloudFront Strict-Transport-Security header** ‚Äî done. A custom `ResponseHeadersPolicy` in [infrastructure/hello_world_frontend_stack.py](infrastructure/hello_world_frontend_stack.py) replaces the AWS-managed `SECURITY_HEADERS` policy, reproducing its four headers (X-Content-Type-Options, X-Frame-Options, Referrer-Policy, X-XSS-Protection) and adding HSTS (`max-age=31536000`, `include_subdomains=True`, `preload=False`) plus a CSP (see "CSP header on the static page" below ‚Äî both now ship from this one policy). `preload` is intentionally off until a custom domain exists, since the preload list is keyed on the registrable domain, not a shared `*.cloudfront.net` host.
- [ ] **Narrow the CDK bootstrap permissions** ‚Äî the default `cdk bootstrap` creates a `CloudFormationExecutionRole` with `AdministratorAccess`. Any identity that can `sts:AssumeRole` into the deployment roles (by default, any principal in the account) can do anything in the account during deploy. Fine for a solo-dev laptop, a headache for organizations. Fix path: re-bootstrap with `cdk bootstrap --custom-permissions-boundary <POLICY_NAME>` so CFN can do anything inside the boundary but can't escape it (e.g., can't attach `AdministratorAccess` or create roles that bypass the boundary). At the org level, use SCPs via AWS Organizations to prevent tampering with the boundary. Restrict who can assume `DeploymentActionRole` to the CI role + named humans. **Sequence this before the Deploy workflow above** ‚Äî once CI gets credentials that can assume the bootstrap roles, the admin default becomes a real blast radius.
- [ ] **Branch protection enforced, not routinely bypassed** ‚Äî branch protection rules exist on `main` (required status checks from CI), but `git push` reports `Bypassed rule violations for refs/heads/main`, meaning maintainers routinely override the gate. For production, each bypass should be an audited exception with a recorded rationale, not a normal merge workflow ‚Äî otherwise the gate exists only to slow down non-maintainers. Tighten via GitHub branch protection settings: require status checks to pass for *administrators* too, require pull-request reviews, and dismiss stale reviews on new commits. The signal "maintainer pushes directly to main" should be unusual enough that each occurrence triggers a question.
- [ ] **Enforce TLS 1.2+ minimum on both edges** ‚Äî the CloudFront distribution and API Gateway both currently sit on AWS-managed default certificates (`*.cloudfront.net` and `*.execute-api.{region}.amazonaws.com`), which pin the TLS floor at **TLS 1.0**. Verified empirically: `curl --tls-max 1.0 https://<dist>.cloudfront.net` and the equivalent against the execute-api endpoint both complete a full handshake. The CDK code in [hello_world_frontend_stack.py](infrastructure/hello_world_frontend_stack.py) sets `TLS_V1_2_2021` but AWS silently overrides it whenever `CloudFrontDefaultCertificate: true`. The cdk-nag rule `AwsSolutions-CFR4` correctly flags this and is intentionally suppressed at the stack level. **Fix path:** acquire a domain, provision an ACM certificate (CloudFront cert must live in us-east-1, API Gateway custom-domain cert lives in the API's region), attach as `viewer_certificate` / `apigateway.DomainName`, then set the strongest matching `securityPolicy` (e.g. `SecurityPolicy_TLS13_2025_EDGE` for an edge-optimized API Gateway domain, `SecurityPolicy_TLS13_1_3_2025_09` for a regional one, and `TLSv1.2_2021` minimum on CloudFront). Once the custom domain is wired and verified, remove the CFR4 suppression. Also reconsider whether the API needs to remain `EDGE` ‚Äî CloudFront already fronts it, so making the backend `REGIONAL` removes the redundant edge layer and unlocks the regional `securityPolicy` set (which includes post-quantum and PFS variants).

- [ ] **Contributing guide** ‚Äî `CONTRIBUTING.md` with fork/branch/PR workflow and pre-commit setup instructions

## Service-level hardening

Per-service hardening items grouped by AWS service so each block can be tackled in isolation. Items deferred for "no direct CDK construct" or "would require restructuring upstream data" reasons are flagged.

### API Gateway

- [ ] **APIGW-layer request validation** ‚Äî currently validation runs inside Lambda via `enable_validation=True` on the resolver. APIGW-level `request_validator_options` rejects malformed requests at the gateway before they hit Lambda billing. Cheaper for adversarial traffic.
- [ ] **Resource policy on the REST API** ‚Äî restrict invocation to specific source IPs, VPCs, or AWS accounts without needing a full authorizer. Useful for partner-facing or internal-only APIs.
- [~] **Close the CloudFront-bypass window on `execute-api.{region}.amazonaws.com`** ‚Äî *option (a) done; option (b) open.* A REGIONAL WAF WebACL is now associated with the Prod stage in [infrastructure/hello_world_app.py](infrastructure/hello_world_app.py) (`_attach_regional_waf`), mirroring the four managed threat rule groups from the CLOUDFRONT-scoped ACL via the shared `build_managed_threat_rules` helper in [infrastructure/nag_utils.py](infrastructure/nag_utils.py). The rate-based rule is intentionally omitted on the origin (every request arrives from a CloudFront edge IP, so IP aggregation would penalise legitimate funnelled traffic; origin volume is bounded by stage throttling + reserved concurrency instead). This closes the *managed-rule* bypass but the origin still **accepts** direct callers ‚Äî for full lockdown, layer option (b): CloudFront injects a Secrets-Manager secret header and an API Gateway resource policy denies any request without it (requires a rotation Lambda + a custom CloudFront origin request policy). `AwsSolutions-APIG3` and the NIST/PCI `APIGWAssociatedWithWAF` suppressions are retired.
- [~] **Custom domain + endpoint export + Mutual TLS** ‚Äî out of scope for the sample app: custom domain depends on owning a domain ([TLS item](#security)); MTLS only justifies its complexity once you have partners. *Canary deployments are now implemented* (CodeDeploy traffic shifting on the Lambda alias ‚Äî see the Lambda section below).

### Lambda

- [ ] **Dead Letter Queue (DLQ)** ‚Äî already covered by [Observability](#observability). Lower priority for the synchronous API path; required if you ever add async invokes (EventBridge, SNS, S3 events).
- [x] **Reserved concurrency** ‚Äî `reserved_concurrent_executions=100` is set on `HelloWorldFunction` in [infrastructure/hello_world_app.py](infrastructure/hello_world_app.py), capping how much of the account concurrency pool a runaway loop or spike on `/hello` can consume. `NIST.800.53.R5-LambdaConcurrency` / `HIPAA.Security-LambdaConcurrency` suppressions retired (the same rule IDs in `CDK_LAMBDA_SUPPRESSIONS` stay ‚Äî they cover CDK-managed singleton Lambdas whose concurrency isn't configurable).
- [x] **Async failure destination on the AwsCustomResource provider Lambda** ‚Äî implemented via `attach_async_failure_destination()` in [infrastructure/nag_utils.py](infrastructure/nag_utils.py). The provider singleton (which CloudFormation invokes asynchronously during stack lifecycle events) writes failed invocation envelopes to an SQS DLQ (CMK-encrypted, 14-day retention, SSL-enforced) so post-mortem evidence isn't lost when a CR crashes through Lambda's two automatic async retries. The HelloWorldFunction itself remains synchronous (API Gateway proxy) and intentionally has no DLQ.
- [x] **Canary deployments (CodeDeploy traffic shifting)** ‚Äî the API integrates with a `live` Lambda alias and a `codedeploy.LambdaDeploymentGroup` shifts traffic onto each new version with automatic rollback on a canary alias-errors alarm (`_attach_canary_deployment` in [infrastructure/hello_world_app.py](infrastructure/hello_world_app.py)). Environment-gated: `CANARY_10PERCENT_5MINUTES` in prod, `ALL_AT_ONCE` in dev/ephemeral. See [README "Deployment safety"](README.md#deployment-safety-canary-lambda--appconfig-rollback).
- [ ] **SnapStart for Python** ‚Äî Python SnapStart launched in November 2024. Roughly 70% cold-start reduction at the cost of a one-time init snapshot. Worth enabling if cold-start latency becomes a UX issue.
- [ ] **Lambda Insights** ‚Äî extension-based enhanced metrics (CPU time, memory utilization, init duration, network bytes). One-line CDK setting (`insights_version=lambda.LambdaInsightsVersion.VERSION_X`); ~$0.50/month per function for the metric stream.

### DynamoDB

The table and its dedicated CMK live in their own stack, [`HelloWorldDataStack`](infrastructure/hello_world_data_stack.py), so their lifecycle is independent of the stateless compute (CDK best practice: keep stateful resources in their own stack). ### Athena

- [x] **Bytes-scanned-per-query data usage control** ‚Äî `bytes_scanned_cutoff_per_query=1 GiB` is set on the access-logs workgroup in [infrastructure/hello_world_frontend_stack.py](infrastructure/hello_world_frontend_stack.py); Athena cancels any query that would exceed it, capping per-query cost against a forgotten `WHERE`. CDK 2.248's L1 `CfnWorkGroup` doesn't expose that field directly; achieving it requires a property override on the underlying CFN resource. The current setup (`enforce_work_group_configuration=True` + `EncryptionOption=SSE_KMS`) already prevents per-query overrides at this workgroup specifically, which is project-scoped and not account- or region-wide, so for a sample app it's equivalent.

- [ ] **Cost allocation tags on the workgroup** ‚Äî apply tags (Environment, Project, Owner) so Athena query costs roll up cleanly in Cost Explorer.
- [ ] **Partition projection on access-log tables** ‚Äî *Deferred: requires upstream log restructuring.* The Glue tables for `cloudfront_logs` and `s3_access_logs` currently scan every file in the prefix on every query because CloudFront standard v1 logs and S3 server access logs both write to flat key spaces (no `year=YYYY/month=MM/...` directories). Both are significant changes and (a) requires CDK to catch up on standard v2 wiring. Revisit when the v2 path is well-supported.
- [ ] **Athena CloudWatch alarms** ‚Äî `QueryFailed` rate, `ProcessedBytes` per query/per workgroup. Same gap as the broader [CloudWatch alarms](#observability) item.

- [ ] **CloudWatch alarms on BlockedRequests spikes and WebACLCapacityUnits (WCU)** ‚Äî same gap as the broader [CloudWatch alarms](#observability) item. - [x] **WAF logs to S3** ‚Äî both WebACLs (CloudFront-scoped in `HelloWorldWafStack`, regional in `_attach_regional_waf`) deliver access logs to `aws-waf-logs-*` S3 buckets (SSE-S3, 90-day lifecycle) instead of CloudWatch ‚Äî cheaper long-term retention, queryable via Athena. WAF owns the bucket policy (CDK pre-declares the delivery grant + orders the logging config after it; verified on a live deploy). - [x] **WAF Glue/Athena tables** ‚Äî two partition-projected Glue tables (`waf_cloudfront_logs`, `waf_regional_logs`) over the `aws-waf-logs-*` buckets, plus 8 Athena named queries (recent blocked / top blocked IPs / top terminating rules / by country, per WebACL), restore the WAF log analysis the WAF‚ÜíS3 switch retired. See [README "Audit stack and log retention"](README.md#audit-stack-and-log-retention).
- [ ] **WAF logging ‚Äî `redacted_fields` and `logging_filter`** ‚Äî per the [`CfnLoggingConfiguration` reference](https://docs.aws.amazon.com/cdk/api/v2/python/aws_cdk.cfn_property_mixins.aws_wafv2/CfnLoggingConfigurationMixinProps.html), WAF logs include full request headers, body, and URI by default. If the API ever accepts an `Authorization` header, a session cookie, or a body field with PII, that lands in the WAF logs (now the `aws-waf-logs-*` S3 bucket) unredacted. Wire `redacted_fields=[{single_header: {name: "authorization"}}, {single_header: {name: "cookie"}}, ...]` so they're scrubbed at log-write time. Separately, `logging_filter` lets you drop ALLOW logs and keep BLOCK/COUNT/CAPTCHA so log volume is proportional to threat traffic ‚Äî relevant once paid traffic actually hits the WAF.
- [ ] **Pin AMR managed-rule-group versions** ‚Äî the WebACL currently uses the floating "default" version of each AMR. Pinning to a specific version (e.g. `Version_2.0` of `AWSManagedRulesCommonRuleSet`) means rule updates from AWS go through your release process rather than landing automatically. Trade-off: less drift, but you have to track AMR change announcements and bump the version manually.
- [ ] **Subscribe to AMR SNS update topics** ‚Äî AWS publishes notifications when managed rule groups change behavior (action shifts, new sub-rules, deprecations). Subscribing means you find out *before* the change lands rather than from a Slack page about a sudden block-rate spike.
- [x] **Add `AWSManagedRulesAnonymousIpList` AMR** ‚Äî implemented at WebACL priority 3 in [hello_world_waf_stack.py](infrastructure/hello_world_waf_stack.py). Blocks Tor exit nodes, hosting providers, and known anonymizing services.
- [ ] **CAPTCHA / Challenge actions on high-risk routes** ‚Äî N/A until login/signup/credential-bearing routes exist. When they do, replacing a `Block` action on a rate-based rule with `Challenge` (silent JS challenge) or `CAPTCHA` (visible) catches bots without false-positiving real users.
- [ ] **Geo-blocking rule** ‚Äî if you have countries that should never reach the app, a single `geoMatchStatement` rule blocks them at the edge. Free, low WCU. Skip if global traffic is expected.
- [ ] **Bot Control / ATP / ACFP** ‚Äî paid advanced AMRs (Bot Control common ~$10/M requests, targeted higher). Bot Control covers automated browser-based traffic; ATP (Account Takeover Prevention) protects login flows; ACFP (Account Creation Fraud Prevention) protects signup flows. - [~] **AntiDDoSRuleSet** ‚Äî *implemented and deliberately reverted.* The rule group provides L7 anti-DDoS via Challenge + Block, but it carries a $20/month per-WebACL entity activation fee plus $0.15/million requests on top of the standard WAF base ($5/month WebACL + $1/month per rule + $0.60/million inspections). See the AntiDDoS write-up in the README for the longer rationale. Revisit if the workload ever sees enough traffic that AWS's classifier has signal to act on, or when the $20/month is rounding error.

### AppConfig

- [ ] **Gradual deployment + alarm rollback monitor** (*built, opt-in via `-c appconfig_monitor=true`; off by default*) ‚Äî the gradual strategy (LINEAR 25%/step over 10 min, 5-min bake) + CloudWatch alarm monitor that auto-rolls-back a bad flag rollout is wired in [`infrastructure/hello_world_app.py`](infrastructure/hello_world_app.py) (`_attach_appconfig_rollback_monitor`), gated behind the `appconfig_monitor` context flag. The observable signal always ships: the handler emits a `FeatureFlagEvaluationFailure` EMF metric on the feature-flag fallback path (a bad config is caught by the fallback and returns 200, so it produces no Lambda error ‚Äî that custom metric is the only signal). **Why it's off by default:** an AppConfig deployment monitor rolls back when its alarm is in `ALARM` **or `INSUFFICIENT_DATA`** ([monitoring-deployments.html](https://docs.aws.amazon.com/appconfig/latest/userguide/monitoring-deployments.html)). On a cold CFN stack the metric has never reported and a fresh alarm starts `INSUFFICIENT_DATA`, so AppConfig aborts the very first deployment ‚Äî the stack can never reach `CREATE_COMPLETE` (proven on a live deploy, even with `treat_missing_data=NOT_BREACHING`). **Adoption sequence (do NOT set the flag on the cold deploy):** (1) `make deploy` with the default (all-at-once, no monitor) so the stack reaches `CREATE_COMPLETE` and the Lambda starts reporting the metric; (2) `make deploy-appconfig-monitor` (guarded ‚Äî refuses unless the stack already exists in an updatable `*_COMPLETE` state) for ongoing flag changes. To persist it across plain deploys, also set `"appconfig_monitor": true` in `cdk.json` after step 1. See [README "Deployment safety"](README.md#deployment-safety-canary-lambda).
- [ ] **AWS AppConfig Lambda extension layer** ‚Äî per [Using AWS AppConfig Agent with AWS Lambda](https://docs.aws.amazon.com/appconfig/latest/userguide/appconfig-integration-lambda-extensions.html), the extension caches configurations in-process, polls in the background, and serves them via `localhost:2772` ‚Äî reducing both AppConfig API spend and cold-start latency. Powertools' `AppConfigStore` does in-memory caching too, so the gain over the current setup is smaller than for raw API users; still the AWS-recommended pattern for high-throughput Lambdas. One-line addition: `_lambda.LayerVersion.from_layer_version_arn(...)` with the regional AppConfig extension ARN, then point Powertools at the localhost endpoint.

### Frontend / browser

- [~] **CSP header on the static page** ‚Äî *baseline done, hardening open.* A `Content-Security-Policy` now ships from the custom `ResponseHeadersPolicy` in [infrastructure/hello_world_frontend_stack.py](infrastructure/hello_world_frontend_stack.py) (paired with HSTS in [Security](#security), from the same policy). It locks `default-src`/`object-src 'none'`/`base-uri`/`frame-ancestors 'none'` and scopes `script-src`/`connect-src` to the exact AWS endpoints (RUM client + data plane, Cognito Identity, and this API's specific `{id}.execute-api.{region}` host ‚Äî not a region wildcard). It still carries `'unsafe-inline'` on `script-src`/`style-src` because `frontend/index.html` ships inline `<script>`/`<style>` blocks (the RUM bootstrap shares state with the body handler via synchronous head execution). To reach a strict nonce/hash CSP, externalize the two inline scripts into `'self'`-served `.js` files first ‚Äî CloudFront can't mint per-response nonces and static hashes break on every edit.
- [ ] **No SRI on the RUM client script** ‚Äî `frontend/index.html` loads `https://client.rum.us-east-1.amazonaws.com/1.x/cwr.js` over a major-version-floating URL so AWS-published security patches reach the browser without a redeploy. The trade-off: a fixed `integrity=` SRI hash would break on the first patched release. The trust model here is "AWS-served domain over TLS 1.2+" rather than pinned content hash. If your fork pins to a specific patch version (e.g., `1.21.0`), add the matching SRI hash so the browser refuses tampered scripts; otherwise document the trade-off explicitly so a future review doesn't re-flag the absence of `integrity=`.
